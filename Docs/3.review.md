## 1. 버그 / 잠재적 오류

### BUG-01 [Critical] CanMove가 점유 상태를 체크하지 않음

**파일**: `QubeBlock.cs:138-142`

```csharp
public bool CanMove(Vector2Int direction)
{
    Vector2Int newPos = position + direction;
    return IsPositionValid(newPos, currentCells, checkOccupancy: false);
}
```

`checkOccupancy: false`로 호출하므로, 블록이 이미 배치된 셀 위로 이동할 수 있음.
`CanPlace()`는 점유를 체크하지만, 이동 시 블록이 시각적으로 다른 블록 위에 겹쳐 표시됨.

**영향**: 플레이어가 블록을 점유된 셀 위로 이동시킨 후 배치하면 `CanPlace()`에서 거부되지만, 시각적 혼란 발생.

**수정 제안**:
```csharp
return IsPositionValid(newPos, currentCells, checkOccupancy: true);
```
또는 이동은 허용하되, 배치 불가 위치일 때 시각적 피드백(반투명, 빨간색 등) 제공.

### BUG-02 [Critical] nextBlockId가 게임 재시작 시 리셋되지 않음

**파일**: `QubeBlock.cs:14`

```csharp
private static int nextBlockId = 0; // Static counter for unique block IDs
```

`NewGame()` 호출 시 `QubeBlock.nextBlockId`는 리셋되지 않음. 게임을 여러 번 재시작하면 blockId가 계속 증가하여, `FindQuadsRespectingBlocks()`의 블록 그룹화에서 이전 게임의 ID 범위와 중복될 가능성은 없지만, 의도와 다르게 동작할 수 있음.

**영향**: 현재 동작에 직접적 버그는 아니지만 (그리드가 클리어되므로), 장기 세션에서 int 오버플로 가능성. 또한 디버그 로그에서 blockId가 0부터 시작하지 않아 혼란.

**수정 제안**: `QubeGameManager.NewGame()`에서 리셋 메서드 호출:
```csharp
QubeBlock.ResetBlockIdCounter();
```

### BUG-04 [Major] RemoveQuads 코루틴과 IncrementTurn의 동기화 문제

**파일**: `QubePulseSystem.cs:66-71`

```csharp
List<QubeQuad> quadsToRemove = trackedQuads.Where(q => q.turnTimer >= PULSE_INTERVAL).ToList();
if (quadsToRemove.Count > 0)
{
    StartCoroutine(RemoveQuads(quadsToRemove));
}
```

`RemoveQuads`는 코루틴 (0.3초 대기 포함)이지만, `IncrementTurn()`은 동기 메서드처럼 진행됨. 소거 코루틴이 완료되기 전에 `HighlightQuads()`가 호출되어, 소거 중인 Quad가 여전히 `trackedQuads`에서 하이라이트될 수 있음.

실제로는 `RemoveQuads` 내에서 `trackedQuads.Remove()`가 `yield` 전에 실행되므로 현재는 동작하지만, `WaitForSeconds(0.3f)` 중에 다른 턴이 처리되면 상태 불일치 가능.

**영향**: `isProcessingTurn` 플래그로 중복 호출은 방지되지만, 코루틴 기반 소거와 동기 턴 처리의 구조적 불일치.

**수정 제안**: `IncrementTurn` 자체를 코루틴으로 변경하거나, 소거를 동기 처리.

### BUG-05 [Major] QubeQuad.Contains()가 O(n) 선형 탐색

**파일**: `QubeQuad.cs:59-62`

```csharp
public bool Contains(Vector2Int cell)
{
    return cells.Contains(cell);  // List<Vector2Int>.Contains → O(n)
}
```

`IsAdjacentTo()`에서 `this.cells × 4방향 × other.Contains()` = O(n × m) 호출.
`ProcessNewQuad`에서 모든 기존 Quad와 비교하므로 O(quads × n × m).

**영향**: 현재 그리드(12×9)에서는 문제없지만, Quad가 많아지면 성능 저하 가능. 또한 이 메서드가 `OverlapsWith`, `IsAdjacentTo`, `CanMergeWith` 등 여러 곳에서 반복 호출됨.

**수정 제안**: `QubeQuad`에 `HashSet<Vector2Int> cellSet` 캐시 추가:
```csharp
private HashSet<Vector2Int> cellSet;
public bool Contains(Vector2Int cell) => cellSet.Contains(cell);  // O(1)
```

### BUG-07 [Minor] CanPlaceAnywhere가 모든 회전 상태를 고려하지 않음

**파일**: `QubeBlock.cs:149-164`

현재 회전 상태(`currentCells`)로만 배치 가능 여부를 판단. 블록을 회전하면 배치 가능한 경우가 있지만, 현재 회전 상태에서만 검사하므로 조기 게임오버 판정 가능.

**영향**: 회전하면 배치 가능한데 게임오버가 되는 경우 발생 가능.

**수정 제안**: 4가지 회전 상태 모두 검사:
```csharp
public bool CanPlaceAnywhere()
{
    Vector2Int[] testCells = (Vector2Int[])currentCells.Clone();
    for (int rot = 0; rot < 4; rot++)
    {
        for (int y = 0; y < HEIGHT; y++)
            for (int x = 0; x < WIDTH; x++)
                if (IsPositionValid(new Vector2Int(x, y), testCells, true))
                    return true;
        // 90도 회전
        for (int i = 0; i < testCells.Length; i++)
            testCells[i] = new Vector2Int(testCells[i].y, -testCells[i].x);
    }
    return false;
}
```

## 2. 성능

### PERF-01 [Major] 셀당 Canvas + CanvasGroup 동적 생성

**파일**: `QubeCell.cs:83-104`

```csharp
Canvas canvas = textObj.AddComponent<Canvas>();
canvas.overrideSorting = true;
canvas.sortingOrder = 1000;

CanvasGroup canvasGroup = textObj.AddComponent<CanvasGroup>();
```

108개 셀 × (Canvas + CanvasGroup) = 216개 추가 컴포넌트. Canvas는 Unity의 배치(batch) 경계를 생성하므로 드로우콜 증가.

**영향**: 모바일 타겟에서 드로우콜 108+ 증가 가능.

**수정 제안**:
1. 타이머 텍스트를 별도 공유 Canvas 아래에 오브젝트 풀로 관리
2. 활성 Quad의 중앙 셀에만 동적 생성/재활용
3. `overrideSorting` 대신 부모 Canvas의 sortingOrder 활용

### PERF-02 [Major] FindQuadsRespectingBlocks의 4중 루프

**파일**: `QubeQuadDetector.cs:127-219`

```csharp
for (int height = maxH; height >= 2; height--)
    for (int width = maxW; width >= 2; width--)
        for (int startY = minY; startY + height - 1 <= maxY; startY++)
            for (int startX = minX; startX + width - 1 <= maxX; startX++)
```

최악의 경우 (전체 그리드가 하나의 연결 영역): `8 × 11 × 8 × 11 = 7,744` 반복.
각 반복에서 내부 이중 루프(`height × width`)와 `quadCells.Contains()` (O(n)) 호출.

**영향**: 현재 그리드에서는 허용 범위이나, 그리드 크기 증가 시 급격한 성능 저하.

**수정 제안**:
1. `quadCells`를 `HashSet`으로 변경 (Contains O(1))
2. 이미 사용된 셀을 빠르게 스킵하는 `bool[,] usedCells` 배열 추가
3. Greedy 탐색을 블록 경계 기반으로 최적화

## 3. 설계 / 구조

### DESIGN-01 [Major] 비주얼과 로직의 결합

**파일**: `QubeCell.cs`, `QubeBlock.cs`

셀의 게임 상태(`isOccupied`, `blockId`, `clearTimer`)와 비주얼 로직(`Image`, `Border`, `TurnTimerText`)이 하나의 클래스에 혼재.

**영향**:
- Sprite 시스템 추가 시 QubeCell 수정 범위 큼
- 테스트 시 MonoBehaviour 의존성으로 단위 테스트 어려움
- 상태 변경과 렌더링이 동시에 일어나 디버깅 복잡

**수정 제안**: 데이터와 뷰 분리
```
QubeCellData (Plain C# class): coordinates, isOccupied, blockId, clearTimer
QubeCellView (MonoBehaviour): Image, Border, Timer → QubeCellData 참조
```

### DESIGN-02 [Major] PlacedBlocks 비주얼과 셀 상태의 이중 관리

**파일**: `QubeGrid.cs:156-165`, `QubeBlock.cs:252-269`

배치된 블록의 비주얼은 `PlacedBlocks` 컨테이너의 자식 Image로 존재하고, 셀 상태는 `QubeCell.isOccupied`로 별도 관리. 소거 시 `RemovePlacedBlockVisual()`에서 이름 기반으로 찾아 삭제:

```csharp
string targetName = $"PlacedCell_{coords.x}_{coords.y}";
Transform child = placedBlocksContainer.transform.Find(targetName);
```

**문제점**:
- `Transform.Find()`는 이름 기반 문자열 매칭 → 오류 가능성
- 셀 상태와 비주얼이 별도로 관리되어 동기화 실패 가능
- Quad 소거 시 비주얼 제거가 `grid.SetCellOccupied()` → `RemovePlacedBlockVisual()` 경로로만 동작

**수정 제안**: `QubeCell`이 자신의 배치 비주얼 참조를 직접 보유:
```csharp
public class QubeCell
{
    public GameObject placedVisual;  // 배치된 블록 Image 참조
}
```

`QubeGrid.cellSize`와 `QubeGrid.spacing`을 참조하지 않고 하드코딩. 그리드 크기 변경 시 타이머 위치 오프셋이 어긋남.

**수정 제안**:
```csharp
float cellStep = grid.cellSize + grid.spacing;
```

### QUAL-05 [Minor] PulseSystem 로그 메시지 불일치

**파일**: `QubePulseSystem.cs:69`

```csharp
Debug.Log($"Removing {quadsToRemove.Count} quads that reached 4 turns");
```

`PULSE_INTERVAL = 8`인데 로그에는 "4 turns"로 표기. 이전 값이 남아있는 것으로 보임.

### QUAL-06 [Minor] QubeBlock.SetParent 시 worldPositionStays

**파일**: `QubeBlock.cs:268`

```csharp
cellObj.transform.SetParent(placedContainer, worldPositionStays: true);
```

`worldPositionStays: true`는 월드 좌표를 유지. 블록이 Canvas 직속 자식으로 생성되고, `PlacedBlocks`도 같은 Canvas 자식이므로 정상 동작하지만, 두 컨테이너의 RectTransform 설정이 다르면 위치 오차 발생 가능.

## 7. 긍정적 평가

- **핵심 게임 루프 완성도 높음**: 배치 → 감지 → 추적 → 소거 사이클이 안정적으로 동작
- **Quad 병합 로직 견고**: Overlap/Adjacent/Contained 3가지 케이스를 잘 분류하여 처리
- **ScriptableObject 활용**: 블록 형태를 데이터 주도적으로 관리
- **코드 구조화**: 메서드 분리가 잘 되어 있고 이름이 의도를 명확히 전달
- **Wall Kick 시스템**: 9-offset 테이블 기반으로 안정적인 회전 처리
- **이벤트 기반 점수**: `OnPulse` 델리게이트로 GameManager와 PulseSystem 간 결합도 낮음
- **방어적 프로그래밍**: `isProcessingTurn`, `isPlacingBlock` 등 중복 호출 방지 플래그 적절
